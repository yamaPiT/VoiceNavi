# SW105 ソフトウェア要求仕様書

## 1. 導入

### 1.1 目的・位置づけ
本システム（VoiceNavi）は、1990年代に提唱された「画面に依存しない、対話による安全なナビゲーション」というコンセプトを、現代の生成AI（LLM）技術を用いてブラウザ上のデモンストレーションとして具現化するWebアプリケーションである。
視覚的情報に頼りすぎない空間把握（メンタルマップ形成）や、ドライバーごとのパーソナライズを実現し、ドライバーとAIが自然な言葉で対話する「思考するパートナー」としてのデモンストレーションを提供する。
さらに、本プロジェクトは単なるナビゲーションのデモに留まらず、AIエージェントと人間が協調して並列・自律的にソフトウェアを開発・テストする「DADA (Document and Agent Driven Agile) プロセス」の実証および教育用シミュレータとしての位置づけ・役割も担う。

### 1.2 対象ユーザと適用範囲
- **デモンストレーション操作者/ドライバー役**: PC上でブラウザを開き、マイクを通じてシステムと対話しながら、VoiceNaviの挙動を体験・確認する。
- **適用範囲**: ブラウザ上でのデモアプリケーションであり、特定の指定ルート（鎌倉〜七里ヶ浜ルート等）に限定したナビゲーションと対話を提供する。

### 1.3 参照ドキュメント
- `docs/process/dada_document_guidelines.md`

### 1.4 用語定義
- **メンタルマップ**: ドライバーが周囲の地理的状況を頭の中に描くための空間把握。
- **STT**: Speech-to-Text (音声認識)。
- **TTS**: Text-to-Speech (音声合成)。

## 2. ソフトウェア概要

### 2.1 ソフトウェアの概要
VoiceNaviは、Google Maps上で自動走行する自車アイコンに連動し、LLMを用いてドライバーと自然言語で対話しながら、経路案内や雑談、周辺情報の提供などをシミュレーションするシステムである。

### 2.2 動作環境
- モダンなPC Webブラウザ（Google Chrome, Microsoft Edge等）。マイクおよび音声出力の利用許可が必須。

### 2.3 ユーザー特性・ペルソナおよびドライブの目的
- **ドライバー (ショウヘイ)**: 30代前半・男性。サザンオールスターズのファン。
- **助手席同乗者 (マミコ)**: 20代・女性。ショウヘイのパートナー。
- **音声ナビ (VoiceNavi)**: ショウヘイのクルマに搭載された賢いAIパートナー。メンタルマップの醸成を促し、ドライブをアシストする。
- **ドライブの目的**: デート。最終目的は、出発地の鎌倉駅から自動運転で移動し、七里ヶ浜の有名店「ダブルドアーズ・七里ガ浜本店」できれいな夕日を見ながら夕食を食べること。

### 2.4 前提条件・制約事項・起動と終了
- **前提条件**: クルマは自動運転されている。右折や左折をするときはその旨を発話して乗員を不安にさせない。UI要件として、ドライバに期待する発話文を右側に一覧表示しておくこと。
- **開発制約**: APIキー等のクレデンシャル情報は、直接ソースコードにハードコーディングせず、環境変数等を利用して適切に秘匿化すること。
- **起動と終了**: ユーザーによる「デモ開始」ボタンの押下で開始し、「デモ終了」または到着後のシステム停止により一連のデモンストレーションを終了する。

## 3. 外部インターフェース要件

### 3.1 ユーザーインターフェース (UI)
- **地図表示 (Google Maps)**
  - 自由に拡大縮小（ズームイン・ズームアウト）ができること。
  - 初回ロード時、デモの走行経路全体が画面に収まるよう、適切なズームレベルでなるべく大きく表示すること。
  - ナビゲーションが案内する走行予定経路を「青い路線（ポリライン）」で描画すること。
  - 自車位置を示すアイコンは大きくし、視覚的に目立つようにすること。
  - AIが発話で「ランドマーク」に言及するタイミングで、そのランドマークのアイコンを強調表示（点滅）させること。
- **情報表示・ログエリア**
  - 現在の車速（km/h）を表示すること。
  - ドライバーの音声認識テキストとAIの発話テキストを、発話者が明確に分かる形式（チャットUI等）で画面に表示すること。
- **操作用コンポーネントとドライバーの発話例リスト**
  - 「デモ開始」「中断」「再開」「デモ終了」ボタンを配置し、デモ状態を制御可能とすること。
  - 自律発話トグルスイッチ: ONとOFFを両端に配置したスライドスイッチとし、視覚的・直感的に機能状態が分かるようにすること。
  - ドライバーの発話例リスト: クリック不可の参考情報として「今どのあたり？」「左に何が見える？」「サザンかけて」「お腹すいた」「前のトラックが煙たい」の5点を右側に表示すること。

### 3.2 外部APIインターフェース
- **フロントエンド・バックエンド間**: REST API (Node.js/Express等による BFF: Backend for Frontend 層)。
- **地図・経路**: Google Maps API (フロントエンドから利用)。
- **音声入力 (STT)**: Web Speech API (フロントエンド・ブラウザ標準) またはそれに準拠するAPI。
- **意図抽出・対話 (LLM)**: Google Gemini API 等 (BFF経由)。
- **音声出力 (TTS)**: Google Cloud TTS API (BFF経由)。

### 3.3 ハードウェア・通信インターフェース
- TCP/IP通信によるインターネットへの接続（外部API連携に必須）。
- PCのマイク入力およびスピーカー出力。

## 4. 機能要件

### [REQ-FN-01] 経路案内と自動運転・車両制御連携
- **入力・処理・出力**:
  - **出発地**: 鎌倉駅前 (35.318906404151114, 139.55127576885596)
  - **目的地**: 七里ヶ浜海岸駐車場 鎌倉側 (35.30432449943571, 139.51450647136232)
  - **速度制御**: 最高速度は50km/hとし、なめりかわ交差点 (35.30928591388183, 139.54531628556066) 通過時は時速10kmに減速する。到着時に停止(0km/h)し案内を発話する。加速・減速は3秒間で補間する。
  - **周辺情報・位置計算**: 進行方向（Heading方位角）に基づき、周辺ランドマークの「右手」「左手」「前方」「後方」を数学的に計算しLLMに提供する。若宮大路を南下中は「前方に海」、134号線を西行中は「左手に海」と動的判定する。
  - **メンタルマップ名所要件**: 鶴岡八幡宮、由比ヶ浜、材木座海岸、Good Mellows、稲村ヶ崎（江ノ島と富士山・夕日）、七里ヶ浜、ダブルドアーズ・七里ガ浜本店等の地理・文脈情報をLLMで活用する。
  - **視程制限 (250mルール)**: ランドマークが直線距離250m以上離れている場合は「しばらく行くと〜があります」と距離感を区別する。
- **検証条件（Acceptance Criteria）**:
  - デモ開始後、ルートに沿って最高50km/hで移動し、なめりかわ交差点で10km/hまで減速し、目的地で完全に停止(0km/h)すること。
  - LLMに送信されるプロンプト内に、現在の進行方向に基づいた「前後左右」の相対位置情報と「海の位置」が正しく付与されていること。
- **異常系（自己修復要件）**:
  - 経路計算APIがエラーを返した場合、システムは移動を開始せずスクリーンに一時的な警告を表示するかリトライを行うこと。

### [REQ-FN-02] 音声入力および意図抽出機能
- **入力・処理・出力**:
  - ユーザーの自然発話（STTテキスト）からLLMを用いて意図解析を行い、応答テキストとアクションコマンド（経路案内、雑談、車両操作等）を生成・抽出する。
- **検証条件（Acceptance Criteria）**:
  - STTの入力文字列長が0（無音・空文字）の場合、LLM APIを呼び出さずに無視すること。
  - 1000文字以上の極端に長い文字列（境界値）を受け取っても、システムがパースエラー等でクラッシュしないこと。
- **異常系（自己修復要件）**:
  - **遅延検知**: LLMのAPI応答が遅延（5.0秒以上）した場合、システムは沈黙せず「少し考えさせてください」等の繋ぎ音声を再生し、バックグラウンドで処理を待機するかリトライを行う。
  - **意図抽出エラー回復**: STTは成功したがLLMの応答JSONが破損していた等の場合、「もう一度お願いできますか？」と聞き返すフェーズへ移行（最大3回リトライ）。3回失敗時は初期状態に戻す。

### [REQ-FN-03] 音声出力 (TTS) 機能
- **入力・処理・出力**:
  - LLMが生成した応答テキストを、Google Cloud TTS APIを使用して再生する。
  - SSMLを活用し、適切な抑揚や間（ポーズ）を持たせる設定とする。（頭切れ防止のため300msの無音挿入等を含む）
  - 前バージョンの発話速度からテンポの良い案内のために調整を行い、UI上のチャット表示欄からはSSMLのタグ「`<speak>`等」をクリーンアップして表示する。
- **検証条件（Acceptance Criteria）**:
  - 正式なテキストまたはSSMLがTTS APIに送信され、音声の不自然な途切れやエラーが生じず、正常にオーディオバッファが再生されること。
  - 画面の「発話テキスト表示欄」には、SSMLタグが含まれない純粋なテキストのみが表示されていること。
- **異常系（自己修復要件）**:
  - Google Cloud TTS APIの呼び出しに失敗（ネットワークエラー、認証エラー等）した場合、ブラウザ標準の Web Speech API の Synthesis（SpeechSynthesisUtterance）にフォールバックして処理を中断・クラッシュさせないこと。

### [REQ-FN-04] メンタルマップ（ランドマーク）連携描画機能
- **入力・処理・出力**:
  - LLMが発話内で言及する予定のランドマークに対応するアイコンを、発話のタイミングに合わせて地図上で点滅（強調）表示させる。
- **検証条件（Acceptance Criteria）**:
  - LLMからのJSON出力等にランドマークIDが含まれ、実際に音声再生が始まる時刻と連動して地図上の指定アイコン要素が更新（アニメーション付与）されること。
- **異常系（自己修復要件）**:
  - 発話に指定されたランドマークIDが地図上に存在しない（バグ等）場合でも、描画処理のエラーを握りつぶし、地図のシミュレーションと音声再生は継続されること。

### [REQ-FN-05] コンシェルジュ・車両操作・自律ガイド・ガードレール機能
- **入力・処理・出力**:
  - **店舗の案内制約**: システムからは基本的に店舗名を宣伝しない（例外：目的地の付近である、ユーザーからの直接の要望がある、観光案内として距離250m以内の視程内にある場合のみ。言及時は特徴を併記）。「Good Mellows」等へのカタカナ併記は不可。
  - **名前呼びかけ制限**: ドライバー（ショウヘイさん）への名前による呼びかけは、システム起動後の【最初の発話の1回のみ】とし、2回目以降の発話では名前の呼びかけを行わないこと。
  - **表記の厳密化**: 「滑川」は誤読防止のため「なめりかわ」と表記し、平仮名で統一する。同乗者（マミコさん）を意識しすぎる言及はデモ中1回に制限。重複する案内はしない（直前と同じ情報のスキップ）。フィラー音声（ちょっと待ってね等）は削除しつつテキストには残す。
  - **自律的ガイド**: 自律発話スイッチがONの場合、ユーザーから問いかけがなくても20〜30秒間隔で周辺状況にもとづくお喋りを行う。
  - **特定車両操作**: 「前のトラックが煙たい」に対して `circulation_mode` アクションを実行し、窓を閉めて内気循環にする（システムログに表示）。またLLMは操作を行った旨を必ず発話テキストに含める。「サザンかけて」等の音楽再生には `play_music` アクションを実行。音楽再生時は実際の音は出ず、画面の時速表示左側に音符アイコンと曲名を表示する。これら複数のアクションは配列として同時実行が可能である。
- **検証条件（Acceptance Criteria）**:
  - 音楽再生 (`play_music`) アクションを受信した際、実際のオーディオは出力されず、画面上の「時速表示」の近傍に「音符記号と楽曲名」がUIとして表示されること。
  - 自律的な案内が2回連続で「同一の内容（周辺情報）」にならないように制御されていること。
- **異常系（自己修復要件）**:
  - 法令違反やシステム能力外の要求をユーザーから受けた場合、LLM側で丁寧に回避・拒否する文章を生成すること。
  - 認識不能なアクションIDを受け取った場合、システムは当該アクション制御を無視し、クラッシュさせずにテキストのUI表示と音声発話のみを実行すること。

## 5. 非機能要件

### 5.1 セキュリティとAPIキー秘匿化
- Google Maps, Gemini, Google Cloud TTS等のAPIキーは、ソースコード上に直接記載することを固く禁じる。
- APIキー等のクレデンシャルは、Node.jsバックエンド（BFF層）および環境変数 (`.env`) に厳重に隠蔽し、フロントエンドやネットワークリクエスト上に一切露出させないアーキテクチャとする。

### 5.2 性能効率性と互換性
- **性能効率性**: ブラウザ上でのレンダリング（地図表示やアニメーション）において、著しいフレーム落ちが発生しない設計とする。TTSやLLMのAPI応答時はユーザーがフリーズを感じないよう非同期処理を適切に実装する。
- **互換性**: モダンブラウザ（Chrome最新版等）で動作すること。スマートフォン等での動作は本デモの必須要件外とする。

### 5.3 使用性 (Usability) と信頼性
- デモの制御要素（開始、中断等）や車両状態（車速等）が一目でわかる整理されたUIを提供する。
- 音声合成やネットワークによる一時的な失敗にも耐えうるフォールバック機能を備え（上記REQ-FN-03等）、デモ自体が強制終了しない（高い信頼性・可用性）設計とする。

### 5.4 保守性と移植性 (アーキテクチャ分離)
- Agentic Architectureとして、STT・LLM解析・TTS・地図描画・UI制御をそれぞれ独立したモジュール（クラス）に分割し、疎結合とすること。
- 各モジュールは依存性を注入・モック化できる形とし、自動テストを容易に実行できるように設計すること。
