# SW105 ソフトウェア要求仕様書

## 1. 導入
### 1.1 目的
本システム（VoiceNavi）は、1990年代に提唱された「画面に依存しない、対話による安全なナビゲーション」というコンセプトを、現代の生成AI（LLM）技術を用いてブラウザ上のデモンストレーションとして具現化するWebアプリケーションである。
視覚的情報に頼りすぎない空間把握（メンタルマップ形成）や、ドライバーごとのパーソナライズを実現し、ドライバーとAIが自然な言葉で対話する「思考するパートナー」としてのデモンストレーションを提供する。
さらに、本プロジェクトは単なるナビゲーションのデモに留まらず、AIエージェントと人間が協調して並列・自律的にソフトウェアを開発・テストする「DADA (Document and Agent Driven Agile) プロセス」の実証および教育用シミュレータとしての役割も担う。

### 1.2 対象ユーザと適用範囲
- **デモンストレーション操作者/ドライバー役**: PC上でブラウザを開き、マイクを通じてシステムと対話しながら、VoiceNaviの挙動を体験・確認する。
- **適用範囲**: ブラウザ上でのデモアプリケーションであり、特定の指定ルート（鎌倉〜七里ヶ浜ルート等）に限定したナビゲーションと対話を提供する。

### 1.3 参照ドキュメント
- `docs/process/dada_document_guidelines.md`

### 1.4 用語定義
- **メンタルマップ**: ドライバーが周囲の地理的状況を頭の中に描くための空間把握。
- **STT/TTS**: Speech-to-Text (音声認識) / Text-to-Speech (音声合成)。

## 2. ソフトウェア概要
### 2.1 ソフトウェアの概要
VoiceNaviは、Google Maps上で自動走行する自車アイコンに連動し、LLMを用いてドライバーと自然言語で対話しながら、経路案内や雑談、周辺情報の提供などをシミュレーションするシステムである。

### 2.2 動作環境
- モダンなPC Webブラウザ（Google Chrome, Microsoft Edge等）。マイクおよび音声出力の利用許可が必須。

### 2.3 ユーザー特性・ペルソナおよびドライブの目的
- **ドライバー (ショウヘイ)**: 30代前半・男性。サザンオールスターズのファン。
- **助手席同乗者 (マミコ)**: 20代・女性。ショウヘイのパートナー。
- **音声ナビ (VoiceNavi)**: ショウヘイのクルマに搭載された賢いAIパートナー。メンタルマップの醸成を促し、ドライブをアシストする。
- **ドライブの目的**: デート。最終目的は、出発地の鎌倉駅から自動運転で移動し、七里ヶ浜の有名店「ダブルドアーズ・七里ガ浜本店」できれいな夕日を見ながら夕食を食べること。

### 2.4 前提条件・制約事項
- クルマは自動運転されている。右折や左折をするときはその旨を発話して乗員を不安にさせない。
- APIキー等のクレデンシャル情報は、直接ソースコードにハードコーディングせず、環境変数等を利用して適切に秘匿化すること。
- UI要件として、ドライバに期待する発話文を右側に一覧表示しておくこと。

## 3. 外部インターフェース要件
### 3.1 ユーザーインターフェース (UI)
- **地図表示 (Google Maps)**
  - 自由に拡大縮小（ズームイン・ズームアウト）ができること。
  - 初回ロード時、デモの走行経路全体が画面に収まるよう、適切なズームレベルでなるべく大きく表示すること。
  - ナビゲーションが案内する走行予定経路を「青い路線（ポリライン）」で描画すること。
  - 自車位置を示すアイコンは大きくし、視覚的に目立つようにすること。
  - AIが発話で「ランドマーク」に言及するタイミングで、そのランドマークのアイコンを強調表示（点滅）させること。
- **情報表示・ログエリア**
  - **現在の車速**（km/h）を表示する。
  - **対話テキスト**: ドライバーの音声認識テキストとAIの発話テキストを、どちらの話者か明確に分かる形式（チャットUIなど）で画面に表示する。
- **操作用ボタンとドライバーの発話例リスト**
  以下4つのボタンを配置し、デモ状態を制御する。
  - **デモ開始**: スタート位置からデモ走行と音声案内を開始する。
  - **中断**: 現在の時点での車両移動と発話を一時停止する。
  - **再開**: 中断した時点・位置からデモを再開する。
  - **デモ終了**: デモ状態を終了し、初期状態にリセットする。
  - 右側に配置されたリストは「ドライバーの発話例（参考）」とし、クリックしても自動入力や発話を行わないように変更すること（ユーザーの誤操作防止のため）。内容も「今どのあたり？」「左に何が見える？」「サザンかけて」「お腹すいた」「前のトラックが煙たい」の5点に絞る。

### 3.2 外部APIインターフェース
- **フロントエンド・バックエンド間**: REST API (Node.js/Express等による BFF: Backend for Frontend 層)
- **地図・経路**: Google Maps API (フロントエンドから利用)
- **音声入力 (STT)**: Web Speech API (フロントエンド・ブラウザ標準)
- **意図抽出・対話 (LLM)**: LLM API (Google Gemini API 等、BFF経由)
- **音声出力 (TTS)**: Google Cloud TTS API（BFF経由。モデル: `Neural2`、日本語、アナウンサー風の女性キャラクター設定）

## 4. 機能要件

### 4.1 経路案内と自動運転・車両制御
- **経路座標定義**:
  - デモの出発地は「鎌倉駅前 (35.318906404151114, 139.55127576885596)」とする。
  - 目的地は「七里ヶ浜海岸駐車場 鎌倉側 (35.30432449943571, 139.51450647136232)」とする。
- **走行速度と交差点減速**:
  - デモ中の最高速度は「50km/h」とする。
  - 「なめりかわ交差点 (35.30928591388183, 139.54531628556066)」を通過する際は、時速10kmに減速する。到達する以前から徐々に速度を落とし、通過後に元の速度へ加速していくこと。
  - 目的地に到達した時点で「停止（0km/h）」し、システムが自動的に目的地到着の案内をLLMで生成・発話すること。
  - 加速・減速アニメーション：0km/hと50km/hの間を「3秒間」で速度変化するように計算。
- **経路上のメンタルマップ情報**: 
  - 鎌倉駅の北直ぐの所に「鶴岡八幡宮」。参道は鶴岡八幡宮から海の方向に伸びている。
  - 滑川の西側が由比ヶ浜、東側が材木座海岸。遠浅な海水浴場として有名。
  - 滑川交差点から、由比ヶ浜沿いに西へ走ると、ハンバーガーの有名店「Good Mellows」がある。「Good Mellows」からさらに西へ行くと「稲村ヶ崎」。
  - 稲村ヶ崎からは江ノ島と富士山が見えて夕日が極めて綺麗。
  - 稲村ヶ崎を超えると七里ヶ浜（「日本の渚100選」、サーフィンメッカ、おしゃれなカフェが並ぶ湘南の定番ドライブデートスポット）。
  - ゴールの七里ヶ浜駐車場近くに「ダブルドアーズ・七里ガ浜本店」が存在する。
- **車両の相対位置に基づく動的案内**:
  - 車両の現在の走行位置および進行方向（Heading方位角）から、周辺ランドマークの「右手」「左手」「前方」「後方」を数学的に正確なアルゴリズムで計算すること。（バグ修正：左右まちがいの解消）
  - **250m視程ルール**: クルマからの視界・視程は「半径250mまで」とする。直線距離が250mより遠い目的地・ランドマークについては「前方に〜見えます」とは言わず、「しばらく行くと〜があります」と距離感を区別してLLMに発話させること。
  - コンテキストプロンプトにより、発話中にクルマが進行して位置関係が乱れないよう、相対案内は早めに手短に済ませること。
- **検証条件（Acceptance Criteria）**:
  - ルート移動による速度変化が行われ、目的地点で完全に停止すること。
  - LLMに送信されるプロンプト内に、進行方向を基準とした「前後左右」の相対位置情報が正しく付与されていること。

### 4.2 音声入力および意図抽出機能
- **機能概要**: ユーザーの自然な発話からLLMを用いて意図解析を行い、応答テキストとアクションコマンド（経路案内、雑談、車両操作等）を生成・抽出する。
- **検証条件（Acceptance Criteria）と境界値**:
  - STTの入力文字列長が0（無音・空文字）の場合、LLMAPIを呼び出さず無視すること。極端に長い文字列（1000文字以上の境界値）でもシステムがパースエラーでクラッシュしないこと。
- **【Autonomous Feedback Loop (自己修復要件)】**:
  - **遅延検知**: LLMのAPI応答が遅延（例: 5.0秒以上）した場合、システムは沈黙せず「少し考えさせてください」等の繋ぎ音声を再生しバックグラウンドで処理を待機するかリトライを行う。
  - **意図抽出エラー回復**: STTは成功したがLLMの応答JSONが破損していた等の場合、「もう一度お願いできますか？」と聞き返すフェーズへ移行（最大3回リトライ）。3回失敗時は初期状態に戻す。

### 4.3 音声出力機能
- **機能概要**: LLMが生成した応答テキストを、Google Cloud TTS APIを使用して再生する。音声モデルは「Neural2」の日本語女性アナウンサーモデルを使用する。
- **スムーズな音声発話とバグ修正**:
  - TTS発話時に「最初の音声が切れてしまう」現象を解消するため、再生バッファやタイミング処理を改善すること。（バグ修正）
  - 現在よりも「少し速い発話速度（rate）」を設定し、テンポの良い案内を実現すること。
  - LLMの応答テキストに SSML (Speech Synthesis Markup Language) 等を活用し、適切な抑揚や間（ポーズ）を持たせること。
  - **UI表示のクリーンアップ**: LLMから返却されたテキスト内に `<speak>` や `<break>` 等のSSMLタグが含まれていたとしても、画面左側の「発話テキスト表示欄」にはそれらタグをすべて除去した純粋なテキストのみを表示すること。（表示バグ修正）
- **検証条件（Acceptance Criteria）と境界値**:
  - 正式なテキスト（または適格なSSML）がTTS APIに送信され、音声の不自然な途切れがなく正常にオーディオバッファが再生されること。
- **異常系（自己修復要件）**:
  - Google Cloud TTS APIの呼び出しに失敗（ネットワーク、認証等）した場合、ブラウザ標準の Web Speech API の Synthesis （SpeechSynthesisUtterance）にフォールバックして処理を中断させないこと。

### 4.4 メンタルマップ（ランドマーク）連携描画
- **機能概要**: LLMが発話内で言及する予定のランドマークに対応するアイコンを、発話のタイミングに合わせて地図上で点滅させる。
- **検証条件（Acceptance Criteria）と境界値**:
  - LLMからの出力（JSON構造内など）にランドマークIDが含まれ、実際に音声再生が始まる時刻と連動して地図上の要素クラスが更新（アニメーション付与）されること。
- **異常系（自己修復要件）**:
  - 指定されたランドマークIDが存在しない場合でも描画処理のエラーを握りつぶし、地図のシミュレーションや音声の再生は継続させること。

### 4.5 コンシェルジュ・車両操作・ガードレール機能
- **機能概要**: ドライブの目的（デート・夕日・夕食）を理解し、それをアシストするように自然発話を行う。また、江ノ島や稲村ヶ崎がサザンオールスターズと関係が深いことを把握しており、ドライバーの要求等に応じてアクションを実行する。
- **特定状況の意図解析と制御**:
  - 「前のトラックが煙たい」等、排気ガスや悪臭に関する発話に対しては、LLMが `circulation_mode` アクションを選択し、「窓を閉めて、内気循環に切り替えるね」といった旨を回答すること。
  - 「Good Mellows」についての言及は、(1)車両が現在その地点より手前を走行中 かつ (2)ユーザーから食欲や店舗紹介に関連する発話があった場合のみ行うように制御すること。
  - 観光案内等の自律的な発話は推奨するが、全く同じ内容の案内はデモ全体を通して最大2回までに制限すること。
- **発話と表記の制約**:
  - UIの表示およびLLMの発話（TTSの読みを含む）において、漢字の「滑川」は誤読を防ぐため平仮名で「なめりかわ」と表記し、統一すること。
  - 「マミコさんが喜ぶ」というような明示的に同乗者を意識し過ぎるフレーズは、1回のデモにつき1回だけにとどめ、口説くような案内が過剰にならないよう制御すること。
- **検証条件（Acceptance Criteria）と境界値**:
  - 音楽再生 (`play_music`) アクションを受信した際、実際のオーディオは出力せず、画面上において「音符記号と楽曲名」が**時速表示の左側**に常に表示されるように UIレイアウトを修正すること。
  - システムは搭乗員が不安にならないよう、自動運転における右左折への言及なども対話内に自然に組み込むこと。
- **異常系（自己修復要件）**:
  - 法令違反やシステム能力外の要求は丁寧に回避・拒否すること。
  - 認識不能なアクションIDを受け取った場合、当該アクションは無視してUI上のテキスト表示のみ実行する。

## 5. 非機能要件
### 5.1 セキュリティとAPIキー秘匿化
- Google Maps, Gemini, Google Cloud TTS等のAPIキーは、ソースコード上に直接記載することを固く禁じる。
- APIキー等のクレデンシャルは、Node.jsバックエンド（BFF層）および環境変数 (`.env`) に厳重に隠蔽し、フロントエンドやネットワークリクエスト上に一切露出させないアーキテクチャとする。

### 5.2 アーキテクチャ分離と保守性
- Agentic Architectureとして、STT・LLM解析・TTS・地図描画・UI制御をそれぞれ独立したモジュール（クラス）に分割し、疎結合とすること。
- 各モジュールは依存性を注入・モック化できる形とし、自動テストを容易に実行できるように設計すること。
