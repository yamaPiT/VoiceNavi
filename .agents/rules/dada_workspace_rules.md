# DADA Process Workspace Rules (特別法)

## 1. DADAプロセスの定義と目的
本プロジェクトは「DADA (Document-and-Agent-Driven Agile) プロセス」を適用します。
目的は、AIエージェントの独走を防ぎ、常に「人間が理解・承認できるドキュメント」と「ソースコード」が完全に一致した高品質なソフトウェアを高速に構築することです。

## 2. DADAプロセスの4原則（厳守事項）
### ① ドキュメント絶対主義 (Single Source of Truth)
- **コード先行の禁止**: いかなる実装・修正も、対応する開発ドキュメント（要求仕様書・設計書等）の更新なしに行ってはなりません。
- **UIの限定**: 主な戦場は「エディターウィンドウ（ドキュメント）」です。チャットでの合意は一時的なものとみなし、必ず実ファイル（`docs/artifact/`内）へ書き出し、確定させてください。

### ② 厳格なる承認ゲート
- **工程遷移の制御**: 「要求→設計」「設計→実装」などのフェーズ移行時には、必ずProduct Ownerの明示的な承認を求めてください。
- **独走の防止**: 承認を得るまでは、次のフェーズの作業（コードの実装など）を開始してはなりません。

### ③ アテンション-リセットの遵守
- **コンテキストの洗浄**: フェーズ移行時には、過去のチャット履歴に含まれる議論や推測を情報源から除外してください。これを確実に行うため、承認直後に**必ずスキル `context-reset` を実行**すること。
- **最新ドキュメントとサマリーの再ロード**: 洗浄後の新しいチャットでは、作業を開始する前に**必ず `docs/process/last_phase_summary.md` を読んで前回の文脈を復元し**、その時点で承認されている「最新の開発ドキュメント」のみを真実として参照し直してください。

### ④ 役割の純化
- 各工程において、AIエージェントは指定されたRole（ペルソナ）に完全になりきり、その役割の範囲外の作業を越権して行わないこと。

## 3. 開発文書と管理表の運用
- 開発成果文書は必ず `docs/artifact/` ディレクトリ配下に保存してください。
- **レビュー記録表 (REV101)**: 工程別に `REV101_Requirements.md`, `REV101_Design.md`, `REV101_Test.md` を使用し、指摘と修正結果を記録します。
- **バグ管理表 (BUG101)**: プログラムのバグおよびコードレビューで検出された問題は `BUG101_バグ管理表.md` に集約します。

## 4. エージェントの行動指針
- **自己校正の義務**: 作業完了後、Product Ownerへ提示する前に、必ず「Reviewer Role」へ切り替わり、自己検閲を行ってください。
- **エスカレーション**: 要求の矛盾や技術的な不確実性を検知した場合は、作業を中断して速やかにProduct Ownerへ報告し、指示を仰いでください。
- **オンデマンド・ロード**: ASDoQモデルや規約類は、執筆・レビューの瞬間にのみロードして基準とすること。

## 5. 共通行動指針（全Roleに適用）
以下のルールは、すべてのRole（作成者・レビュアー）に共通で適用されます。各Roleファイルに重複して記述せず、ここに集約します。

### 品質基準のロード
- **通常時**: 文書の執筆・レビュー時には `docs/guidelines/asdoq_checklist.md`（軽量版）をロードし、チェックリストとして使用すること。
- **大幅改訂時**: Product Ownerから「ASDoQ文書品質モデルに基づき厳密にレビューして」等の指示があった場合のみ、`docs/guidelines/asdoq_model_markdown.md`（フル版：例文・違反例付き）をフルロードすること。

### ドキュメントの構成準拠
- 文書の構成は `DADA-Process.md` の「ドキュメント準拠性」ルールに従うこと。Product Ownerからテンプレート指定（IEEE29148等）がある場合は、`docs/templates/` 配下の該当ひな形を優先すること。

### レビュー指摘の記録
- 作成者Roleが修正を行った後は、対応するレビュー記録表（`docs/artifact/REV101_*.md`）に対応結果を記録すること。
- レビュアーRoleが指摘を行う際は、同じレビュー記録表に指摘内容を具体的かつ客観的に記入すること。

## 6. スケール制御（変更規模に応じたフロー最適化）
AIエージェントは、Product Ownerからの指示内容を分析し、以下の基準で**変更スケールを自律的に判定**します。判定結果は `docs/process/last_phase_summary.md` の引き継ぎ事項に記録し、Product Ownerが後から確認できるようにしてください。

| スケール | 判断基準 | 適用フロー |
| :--- | :--- | :--- |
| **Hotfix** | バグ修正のみ。要求・設計の変更を伴わない | Phase 4 → Phase 5 のみ（BUG101に記録） |
| **Minor** | 既存機能の微調整。設計書の変更を伴わない | Phase 1（差分更新）→ Phase 4 → Phase 5 |
| **Major** | 新機能追加や大幅な仕様変更 | 全Phase通過（標準フロー） |

- **判定報告の義務**: AIは判定結果を「本変更は **[Hotfix/Minor/Major]** と判定しました。理由: ～」の形式でProduct Ownerに報告し、異議がなければ該当フローで進行すること。
- **エスカレーション**: 判定に迷う場合は、安全側（Major）として扱い、Product Ownerに判断を仰ぐこと。
