# ADAプロセス V字モデル標準開発ワークフロー

本プロジェクトでは「コード先行」の開発を固く禁じます。新機能の追加やプログラムの改訂を行う際は、専門化されたエージェントと人間が協調する以下の **ADA (Agent-Driven Agile) プロセス（V字モデル）** を厳守して作業を進めてください。

## ワークフローの各ステップ

### ステップ1: 要求定義 (Requirements Definition)
1. **本体作業 (`requirements-engineer`)**:
   - マサから提示された要求の概要を基に、通常は `./docs/process/ada_document_guidelines.md` の目次構造に従い、要求仕様書（`doc/SW105_ソフトウェア要求仕様書.md` 等）を `doc` フォルダ下に作成します。
   - 作成時、各シナリオや機能に対し、**【検証条件（Acceptance Criteria）】**およびエラー時の**【異常系（自己修復・フォールバック要件）】**を必ず定義し、後続のAgentic開発（自動テスト・自律改善）を可能にしてください。
   - マサから「後は自動で行うように」との指示がない限り、要求を深掘りするための「壁打ち（積極的な提案と質問）」を行い、要件を洗練させてください。
   - ※「根本レビュー」の指示があった場合に限り、`docs/process/` 内の重厚なPDF資料（ESPR, IEEE29148等）を読み込み、規格の精神に則った根本的な再構築を行ってください。
2. **レビュー (`requirements-reviewer`)**:
   - 要求定義に関する高度な技術力とASDoQ定石（特に「目的適合性」）に基づき、作成された要求仕様書を深くレビューします。
3. **最終承認 (人間/マサ)**:
   - レビューを通過した要求仕様書に対して、マサの承認（Approval）を得てください。
4. **AIドキュメント生成ルール**:
   - 不要な長文説明は省き、簡潔・明瞭で「読むだけで構造がわかる」フォーマット（Markdown表やMermaid等）を駆使すること。
   - ※Mermaid図を生成する際は、エディタのプレビュー拡張機能で正しく表示させるため、ノード名のダブルクォート囲みやHTML使用禁止など、`project-rules.mdc` の「安全なMermaid構文の徹底」に厳格に従うこと。

### ステップ2: アーキテクチャ設計 (Architecture Design)
1. **本体作業 (`architect`)**:
   - 承認済みの要求仕様書に基づき、開発対象の技術領域（Web・組込み等）に応じた最適な目次構成でアーキテクチャを設計し、アーキテクチャ設計書（`doc/SW205_ソフトウェアアーキテクチャ設計書.md` 等）を `doc` フォルダ下に作成・更新します。全体構造はMermaid図等を用いて視覚化します。
   - ※ 本ADAプロセスでは、システム全体の上位コンテキスト（アーキテクチャ）の合意に注力するため、個別の機能に関する「ソフトウェア詳細設計書（SW305）」の作成・管理は行いません。
2. **レビュー (`architecture-reviewer`)**:
   - アーキテクチャ構築の高度な技術力とASDoQに基づき、設計の妥当性、整合性、保守性を深くレビューします。レビュー指摘があれば設計書を修正します。

### ステップ3: 実装 (Implementation)
1. **詳細設計と自律的実装計画の策定**:
   - AIエージェントが自律的に作成する `docs/Implementation_plan.md` 等の実装手順計画が、実質的な「詳細設計」の役割を担います。
2. **環境の初期化と実装準備**:
   - 承認された設計に基づき、必要なディレクトリ（`src/` 等）の作成、パッケージの初期設定（`npm init` 等）、`.gitignore` の調整などの環境構築を自律的に実行します。
3. **本体作業 (`programmer`)**:
   - 設計書に基づき、高品質なソースコードを実装します。
   - **【重要】** 単に動くコードを書くだけでなく、プログラムの保守性を高めるため、ASDoQの観点を取り入れた「リッチなコメント（目的、背景、複雑なロジックの意図など）」を必ず記述してください。
4. **レビュー (`code-reviewer`)**:
   - 実装の高度な技術力に加え、ASDoQの文書品質モデルの観点をプログラム言語にも適用し、コードのバグやスタイルだけでなく、コメント文の品質やプログラム全体の保守性の高さをレビューします。

### ステップ4: テスト (Testing)
1. **本体作業 (`test-engineer`)**:
   - ESPRにならい、テストを「単体テスト」「ソフトウェア結合」「ソフトウェア総合テスト（適格性確認）」の3段階で捉えます。
   - 「単体テスト」「ソフトウェア結合」については、自動テスト生成ワークフロー（`@[/generate-unit-tests]` 等）を活用し、テストケースの構築・実行・自己修復を自律的かつ自動で行います。
   - **【重要: 総合テスト仕様書・報告書の作成】** 最終的な総合テストについては、必ず要求仕様書（SW105等）の各要件と1対1で対応づけたテスト項目を網羅した「ソフトウェア総合テスト仕様書・報告書」（例: `doc/SWP6_ソフトウェア総合テスト仕様書・報告書.md`）を `doc` フォルダ下に作成・更新してください。
   - テスト実行後は、同ドキュメント内にテスト結果として「チェック印（`[x]` や `✅` など）」を明記し、人間（マサ）が一目で合否を確認できる適格性確認の証跡として完成させてください。
2. **レビュー (`test-reviewer`)**:
   - テスト設計の高度な専門能力を用い、単体・結合テストの網羅性を評価するとともに、作成された「ソフトウェア総合テスト仕様書・報告書（SWP6）」がASDoQ基準を満たし、要求仕様とのトレーサビリティが確保されているか（漏れがないか）、チェック印が正しく付与されているかを厳しくレビューします。
3. **評価と要求改訂 (人間/マサ)**:
   - 全プロセス完了後に、完成した「ソフトウェア総合テスト仕様書・報告書（SWP6）」をマサが確認し、プログラムが要求通りに作られたかを評価します。
   - 評価結果に基づき、完了とするか、あるいはさらなる向上のために「要求を改訂（ソフトウェア要求仕様書の変更からADAプロセスを再始動）」します。

---
**共通ルール**:
- 各本体作業は、完了後に必ず対応する専門レビュアーエージェントに評価させ、指摘事項があれば校正を行ってから次のステップへ進むこと。
- ドキュメント単位で「前回から変更なし」等の省略はせず、単体で「甘美な状態（完璧で自己完結した状態）」を保つこと。
