# DADAプロセス V字モデル標準開発ワークフロー

本プロジェクトでは「コード先行」の開発を固く禁じます。新機能の追加やプログラムの改訂を行う際は、専門化されたエージェントと人間が協調する以下の **DADA (Document and Agent Driven Agile) プロセス（V字モデル）** を厳守して作業を進めてください。

## ワークフローの各ステップ

### ステップ1: 要求定義 (Requirements Definition)
1. **本体作業 ([`requirements-engineer`](../skills/requirements-engineer/SKILL.md))**:
   - マサから提示された要求の概要を基に、通常は `./docs/process/dada_document_guidelines.md` の目次構造に従い、要求仕様書（`doc/SW105_ソフトウェア要求仕様書.md` 等）を `doc` フォルダ下に作成します。
   - 作成時、各シナリオや機能に対し、**【検証条件（Acceptance Criteria）】**およびエラー時の**【異常系（自己修復・フォールバック要件）】**を必ず定義し、後続のAgentic開発（自動テスト・自律改善）を可能にしてください。
   - マサから「後は自動で行うように」との指示がない限り、要求を深掘りするための「壁打ち（積極的な提案と質問）」を行い、要件を洗練させてください。
   - ※「根本レビュー」の指示があった場合に限り、`docs/process/` 内の重厚な規格要約ドキュメント（ESPR, IEEE29148等をMarkdown化したもの）を読み込み、規格の精神に則った根本的な再構築を行ってください。

2. **自己レビューと校正 (Self-Correction as `requirements-reviewer`)**:
   - 本体作業後、エージェントは自律的にレビュアーのペルソナへ切り替わり、ASDoQ定石（特に「目的適合性」）に基づいて要求仕様書を自己精査し、必要に応じて一括で校正します（トークン節約と高速化のため、明示的な他エージェント呼び出しは不要です）。

3. **最終承認 (人間/マサ)**:
   - レビューを通過した要求仕様書に対して、マサの承認（Approval）を得てください。
   - **【厳守】エージェントはここで必ず作業を停止（待機）し、マサからの明示的な合意・承認の返答があるまで、絶対にステップ2（アーキテクチャ設計）へ進んではいけません。**

### ステップ2: アーキテクチャ設計 (Architecture Design)
1. **本体作業 ([`architect`](../skills/architect/SKILL.md))**:
   - **【開始条件】マサによる要求仕様書の承認がチャット履歴上で確認できること。承認がない状態での推測による作業開始は厳禁です。**
   - 承認済みの要求仕様書に基づき、開発対象の技術領域（Web・組込み等）に応じた最適な目次構成でアーキテクチャを設計し、ソフトウェアアーキテクチャ設計書（`doc/SW205_ソフトウェアアーキテクチャ設計書.md` 等）を `doc` フォルダ下に作成・更新します。全体構造はMermaid図等を用いて視覚化します。
   - ※ 本DADAプロセスでは、システム全体の上位コンテキスト（アーキテクチャ）の合意に注力するため、個別の機能に関する「ソフトウェア詳細設計書（SW305）」の作成・管理は行いません。

2. **自己レビューと校正 (Self-Correction as `architecture-reviewer`)**:
   - エージェントは自律的にレビュアーのペルソナへ切り替わり、ASDoQに基づき設計の妥当性、整合性、保守性を自己精査し、一括で校正します。

3. **最終承認 (人間/マサ)**:
   - **初回作成時**（要求仕様書に基づく新規作成時）は、設計書がない状態からの大幅な書き換えとなるため、必ずマサのレビュー・承認（Approval）を得てください。
   - **改訂時**（実装・テスト工程中などに設計変更が発生した際）は、**設計書が大幅に書き換わったときだけ**マサの再承認を得てください。
   - ※ したがって、アーキテクチャの根幹に影響しない一部の修正や軽微な設計変更（インターフェース名の変更、定数追加等）については、エージェントが自律的に設計書を更新し、マサの再承認なしにそのまま作業を継続して構いません。

### ステップ3: 実装 (Implementation)
1. **詳細設計と自律的実装計画の策定**:
   - AIエージェントが自律的に作成する `docs/Implementation_plan.md` 等の実装手順計画が、実質的な「詳細設計」の役割を担います。

2. **環境の初期化と実装準備**:
   - 承認された設計に基づき、必要なディレクトリ（`src/` 等）の作成、パッケージの初期設定（`npm init` 等）、`.gitignore` の調整などの環境構築を自律的に実行します。

3. **本体作業 ([`programmer`](../skills/programmer/SKILL.md))**:
   - 設計書に基づき、高品質なソースコードを実装します。
   - **自律的な設計修正（ドキュメント先行原則）**: 実装を進める中で根本的な設計変更が必要になった場合、**コード修正に先立って**「ステップ2: アーキテクチャ設計」に立ち戻り、設計書を最新状態に更新した上で実装を再開します。なお、単純なコーディングミス（バグ）の修正は設計書の改訂は不要です。また、マイナな設計変更（アーキテクチャの根幹に影響しない軽微な修正）の場合は、設計書を更新した上で人間の再承認なしに作業を継続して構いません。さらに、実装を進める中で要求レベルの不備や変更が必要と判明した場合は、「ステップ1: 要求定義」に立ち戻り、必ず要求仕様書（SW105）を書き換えて、要求の追加・変更・削除を反映させてください（既存の妥当な要求は維持します）。
   - **【重要】** 単に動くコードを書くだけでなく、プログラムの保守性を高めるため、**ファイルヘッダコメント（必須）および全関数への関数ヘッダコメント（必須）**を含む「リッチなコメント（目的、背景、複雑なロジックの意図など）」を必ず記述してください。

4. **自己レビューと校正 (Self-Correction as `code-reviewer`)**:
   - エージェントは自律的にレビュアーのペルソナへ切り替わり、ASDoQの観点をコードに適用して、バグやスタイル、コメント文の品質（保守性）を自己精査し、一括で修正します。

### ステップ4: テスト (Testing)

テストは以下の3段階で実施します。SWP6ドキュメントは「テスト仕様書」と「テスト報告書」を兼ねる文書であり、**まずテスト仕様書としてテスト項目を設計・記入**し、その後**テストを実行して結果をチェックしテスト報告書として完成**させます。

#### 4-a. 単体テスト・ソフトウェア結合テスト
- **本体作業 ([`test-engineer`](../skills/test-engineer/SKILL.md))**:
  - ESPRにならい、自動テスト生成ワークフロー（`@[/generate-unit-tests]` 等）を活用し、テストケースの構築・実行・自己修復を自律的かつ自動で行います。
  - **自律的な設計・実装・要求修正**: テスト実行中にバグを発見した場合は「ステップ3: 実装」へ、根本的な設計不備を発見した場合は「ステップ2: アーキテクチャ設計」へ、要求自体の不備や変更の必要性を発見した場合は「ステップ1: 要求定義」へ自律的に戻り、ドキュメントとコードを更新します。要求定義に戻る場合は、必ず要求仕様書（SW105）を書き換えて要求の追加・変更・削除を反映させてください（既存の適切な要求は維持します）。

#### 4-b. SWP6テスト仕様書の作成（テスト設計フェーズ）
- **本体作業 ([`test-engineer`](../skills/test-engineer/SKILL.md))**:
  1. 要求仕様書（`doc/SW105_ソフトウェア要求仕様書.md` 等）を読み込み、全要件（機能要件：セクション4、非機能要件：セクション5）のIDと検証条件（Acceptance Criteria）を抽出する。
  2. 各要件に対して1対1で対応するテストケースを設計し、`dada_document_guidelines.md` の目次構造【C】に従ってテスト仕様書として `doc/SWP6_ソフトウェア総合テスト仕様書・報告書.md` に記入する。
  3. テストケースはMarkdown表形式で記述し、各テスト項目の「結果」列は**この段階では空欄**とする（テスト仕様書の状態）。
  4. 要件対応マトリクス（全要件ID ↔ テストケースIDの対応表）を作成し、漏れがないことを確認する。

#### 4-c. テスト実行とSWP6テスト報告書の完成（テスト実行フェーズ）
- **本体作業 ([`test-engineer`](../skills/test-engineer/SKILL.md))**:
  1. 4-bで設計した各テストケースを実際に実行する。
  2. 各テスト項目の「結果」列にチェック印（✅ 合格 / ❌ 不合格 / ⚠️ 条件付き合格 / ➖ 対象外）を記入する。
  3. 不合格のテストがあれば、原因を分析し、可能な限り自己修復してから再テストを行う（「ステップ3: 実装」への差し戻しを含む）。
  4. テスト結果総括（合格率サマリー、残課題、推奨事項）を作成し、SWP6をテスト報告書として完成させる。

#### 4-d. 自己レビューと校正 (Self-Correction as `test-reviewer`)
- エージェントは自律的にレビュアーのペルソナへ切り替わり、以下を自己精査します：
  - 要求仕様書の**全要件**がSWP6のテストケースとして漏れなく対応付けられているか（要件カバレッジ100%）
  - テスト結果が**全項目**に記入されているか（空欄がないか）
  - ASDoQ基準適合性（理解容易性、一貫性）
  - 不備があれば一括で修正・補完し、SWP6を完成状態にします。

#### 4-e. 評価と要求改訂 (人間/マサ)
- 全プロセス完了後に、完成した「ソフトウェア総合テスト仕様書・報告書（SWP6）」をマサが確認し、プログラムが要求通りに作られたかを評価します。
- 評価結果に基づき、完了とするか、あるいはさらなる向上のために「要求を改訂（ソフトウェア要求仕様書の変更からDADAプロセスを再始動）」します。要求を改訂してステップ1に戻る場合、必ず要求仕様書（SW105）を書き換えて、追加・変更・削除された要求内容で最新化してください（変更の必要がない既存の要求はそのまま維持します）。

---
**共通ルール（最適化版）**:
- **【自動化・高速化前提】**: エージェントは、各専門スキルや「ASDoQシステム開発文書品質モデル」の原則を**既に熟知している（In-Context）**前提で動作します。作業前に都度 `SKILL.md` や `asdoq-compliance` をファイル読み込みする必要はありません（ただし、各ドキュメントの新規作成時や大幅改訂時には、必ず `docs/process/dada_document_guidelines.md` を読み込んで目次構造を正確に適用してください。小規模な更新時は読み込みを省略して高速化します）。
- **【自律的なペルソナ切り替え（自己レビュー）】**: 各本体作業後は、外部エージェントを明示的に呼び出すのではなく、自身が瞬時に「専門レビュアー」のペルソナに切り替わり、自己校正（Self-Correction）を完了させてから次のステップへ進んでください（高速化のため）。
- **【ドキュメント先行原則（Single Source of Truth）】**: 実装・テスト中に**設計変更を伴う修正**が必要になった場合は、コードを先に修正するのではなく、**必ず先立って該当するドキュメント（SW105要求仕様書、SW205アーキテクチャ設計書等）を最新状態に更新してください**。ただし、単純なコーディングミスの修正は設計書の改訂は不要です。ドキュメントと実装の乖離は許されません。
- ドキュメント単位で「前回から変更なし」等の省略はせず、単体で「甘美な状態（完璧で自己完結した状態）」を保つこと。
- 不要な長文説明は省き、簡潔・明瞭で「読むだけで構造がわかる」フォーマット（Markdown表やMermaid等）を駆使すること。
- ※Mermaid図を生成する際は、エディタのプレビュー拡張機能で正しく表示させるため、ノード名のダブルクォート囲みやHTML使用禁止など、`project-rules.mdc` の「安全なMermaid構文の徹底」に厳格に従うこと。
