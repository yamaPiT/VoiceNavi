# DADAプロセス ドキュメント作成ガイドライン

## 1. 目的と適用範囲
本ガイドラインは、DADA (Document and Agent Driven Agile) プロセスにおいて、AIエージェントが開発ドキュメントを作成・更新するための軽量なルールブックである。
作成対象となるドキュメントは以下の3種類に限定する。
1. **ソフトウェア要求仕様書 (SRS)**
2. **ソフトウェアアーキテクチャ設計書 (Architecture Design)**
3. **ソフトウェア総合テスト仕様書・報告書 (SWP6)**

## 2. エージェントへの基本ルール（Agent Instructions）
* **一貫性と追跡可能性:** 要求(SRS) → 設計(Arch) → テスト(Test) の間で内容に矛盾や飛躍がないこと。
* **出力形式:** 原則としてMarkdown形式で出力し、可読性を高く保つこと。
※ ドキュメント絶対主義、自己校正の手順、記述すべきDADAコア要件（Agentic Testability等）の詳細は、`.agents/rules/dada_workspace_rules.md` に従うこと。

---

## 3. ドキュメント別 標準目次構造

### 【A】 ソフトウェア要求仕様書 (SW105)
本ドキュメントは、要求定義の最終成果物である。後工程（アーキテクチャ設計、総合テスト）で「合否判定」が可能なレベルまで解像度を上げて記述すること。

* **1. 概要**: 本ソフトウェアの存在意義と前提を定義する。
  * 1.1 目的と位置づけ: 解決する課題とシステム全体における役割。
  * 1.2 適用範囲: 対象とするシステムの境界線（スコープ）。
  * 1.3 用語定義: 開発で共通使用するドメイン用語や略語。
* **2. システム全体構成**: 
  * 2.1 システム構成概要: ハードウェアおよび連携する外部システムを含めた全体アーキテクチャ。
  * 2.2 ソフトウェアの構成要素: 管轄する主要なブロックやモジュール群の概要。
* **3. 制約条件と前提事項**: 絶対に遵守すべき境界条件。
  * 3.1 ハードウェア制約: メモリ、CPU、周辺機器など。
  * 3.2 ソフトウェア環境: OS、ミドルウェア、開発言語、通信プロトコル。
  * 3.3 準拠規格: 法令や業界標準。
* **4. 機能要求**: システムが「何をしなければならないか」を正常・異常系の両面から定義する。
  * 4.1 ユースケースとシナリオ: エンドユーザまたは外部システムとのインタラクション。
  * 4.2 機能詳細: 具体的な機能リストと、それぞれの「入力・処理・出力」。
* **5. 外部インタフェース要求**:
  * 5.1 ユーザインタフェース (UI): 画面構成や操作要求。
  * 5.2 ハードウェアインタフェース: 物理デバイスとの通信・制御仕様。
  * 5.3 ソフトウェアインタフェース: 他システムや外部APIとの通信手順。
* **6. 非機能要求（性能・品質等）**: 
  * 6.1 性能・効率性: 応答時間、スループット等の目標値。
  * 6.2 信頼性・安全性: 許容ダウンタイム、例外時のフェールセーフやリカバリ手順。
  * 6.3 保守性・拡張性: 将来の機能追加や変更に対する方針。
  * 6.4 セキュリティ: データ保護、認証対策。
* **7. その他・特記事項**: 未決定事項（TBD）や引き継ぎ事項。

### 【B】 ソフトウェア・アーキテクチャ設計書 (SW205)
SW105を満たすための「静的構造」と「動的振る舞い」を定義し、詳細設計・実装の入力として曖昧さのない状態にすること。

* **1. 概要**:
  * 1.1 目的と位置づけ / 1.2 適用範囲 / 1.3 参照ドキュメント / 1.4 用語定義
* **2. システム構成**:
  * 2.1 システム全体構成: デバイスや外部システムとの境界。
  * 2.2 主たるソフトウェア要素: 主要な役割と構成要素。
* **3. ソフトウェア構成（静的構造）**: 内部モジュール分割と依存関係。
  * 3.1 ソフトウェア全体構成: 階層化・構造化（レイヤ構造、MVC等）の基本方針。
  * 3.2 機能ユニットの定義: 各モジュールの役割と依存関係。**※インターフェース境界での「検証条件」を明記すること。**
* **4. 制御方式（動的振る舞いとリソース割り当て）**:
  * 4.1 メモリ構成とレイアウト: メモリマップ、ROM/RAM、ヒープ/スタック方針。
  * 4.2 ソフトウェア制御方式: 並行処理方式（タスク、排他制御、プロセス間通信）、状態遷移方針。
  * 4.3 性能見積り: リアルタイム性などの定量的な見積もり結果。
* **5. 機能ユニット詳細**: 各機能ユニットが外部に公開するインタフェース（API、引数、戻り値）。
* **6. システムで扱うデータ**: 共有データ構造、グローバル変数、DBスキーマ。
* **7. 例外・異常処理一覧**: ハードウェアエラー、通信エラー等の異常ケースとリカバリ手順。
* **8. その他・特記事項**: 未決定事項や注意事項。

### 【C】 ソフトウェア総合テスト仕様書・報告書 (SWP6)
「テスト仕様書」と「テスト報告書」を兼ねる。**まず計画フェーズで1〜4章を設計**し、テスト実行後に**結果（5〜6章）を追記して報告書として完成**させる。

**【計画フェーズ】**
* **1. テストの概要と目的**:
  * 1.1 目的 / 1.2 参照ドキュメント / 1.3 対象範囲とバージョン
* **2. テスト環境とシステム構成**:
  * 2.1 システム構成 / 2.2 テスト設備・ツール
* **3. テスト方針**:
  * 3.1 合否判定基準 / 3.2 異常系・例外テスト方針（限界値、高負荷、フェールセーフ確認等）
* **4. テストシナリオと項目詳細（計画）**: 
  * SW105の要求事項を網羅するテスト項目リスト（Markdown表: テストID / 対象要件 / シナリオ / 前提条件 / 入力データ / 期待される出力）。

**【実行・報告フェーズ】**
* **5. テスト実行結果**: 自律実行または手動によるテスト結果の追記。
  * 5.1 実行環境の記録 / 5.2 結果詳細（Markdown表: テストID / 実行日時 / 検査結果 / 実際の出力・エラー / 備考）。
* **6. 総合判定と不具合報告**:
  * 6.1 検出された不具合: Fail原因と修正状況 / 6.2 総合判定: 本プロジェクトの品質証明としての最終合否（Pass/Fail）。

## 4. 自己校正のレビュー基準 (Self-Correction Criteria)
エージェントは自律的かつ瞬時に自己精査（Self-Correction）を完結させること。レビュー時の評価基準は `.agents/rules/dada_workspace_rules.md` の「共通行動指針」に従い、通常は `docs/guidelines/asdoq_checklist.md`（軽量版）、詳細校正時は `docs/guidelines/asdoq_model_markdown.md`（フル版）を使用する。
