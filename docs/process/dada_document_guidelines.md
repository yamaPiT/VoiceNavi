# DADAプロセス ドキュメント作成ガイドライン

## 1. 目的と適用範囲
本ガイドラインは、DADA (Document and Agent Driven Agile) プロセスにおいて、AIエージェントが開発ドキュメントを作成・更新するための軽量なルールブックである。
作成対象となるドキュメントは以下の3種類に限定する。
1. **ソフトウェア要求仕様書 (SRS)**
2. **ソフトウェアアーキテクチャ設計書 (Architecture Design)**
3. **ソフトウェア総合テスト仕様書・報告書 (SWP6)**

## 2. エージェントへの基本ルール（Agent Instructions）
* **役割と自己校正（Self-Correction）の完遂:** 各アクティビティでの本体作業後、自身でレビュアーペルソナへ切り替わって自己校正まで完結させること（明示的な外部レビュアー呼び出しを省略して高速化）。
* **一貫性と追跡可能性:** 要求(SRS) → 設計(Arch) → テスト(Test) の間で内容に矛盾や飛躍がないこと。
* **出力形式:** 原則としてMarkdown形式で出力し、可読性を高く保つこと。
* **【重要】DADAコア要件の記述義務:**
    * **Agentic Testability:** すべての機能要件やコンポーネント設計において、後続の自動テスト生成を可能にするための「明確な検証条件（Acceptance Criteria）と境界値」を必ず記述すること。
    * **Autonomous Feedback Loop:** 異常系やエラー発生時にシステムがどう自己修復・フォールバックするかを必ず記述すること。
    * **Living Detailed Design (Code Comments):** ソースコードの実装においては、ファイルヘッダおよび**すべての関数**に対する明示的な関数ヘッダコメントを記述し、コード自体を「生きた詳細設計」として機能させること。

## 3. ドキュメント別 標準目次構造

### 【A】 ソフトウェア要求仕様書 (SRS)
システムが「何をするか（What）」を定義する。
* **1. 導入:** 目的、位置づけ、対象ユーザ、適用範囲、参照ドキュメント、用語定義
* **2. ソフトウェア概要:** ソフトウェアの概要、動作環境、ユーザー特性、前提条件、開発制約、その他の制約事項、起動と終了
* **3. 外部インターフェース要件:** UI、API、ハードウェア・通信インターフェース
* **4. 機能要件:** ユースケースや機能群ごとの入力・処理・出力。**※各機能の末尾に「検証条件」と「異常系（自己修復要件）」を明記すること。**
* **5. 非機能要件:** 性能効率性、互換性、使用性、信頼性、セキュリティ、保守性、移植性

### 【B】 ソフトウェアアーキテクチャ設計書 (Architecture Design)
要求仕様（SRS）を満たすために、システムを「どう作るか（How）」の全体構造を定義する。
* **1. 導入:** 目的、対象範囲、参照する要求仕様書(SRS)
* **2. システムアーキテクチャ:** 全体構成図、技術スタック
* **3. コンポーネント設計:** 各役割と責任。**※インターフェース境界での「検証条件」を明記すること。**
* **4. データアーキテクチャ:** データモデル、ライフサイクル
* **5. 非機能設計:** セキュリティ、エラーハンドリング（フォールバック）方針

### 【C】 ソフトウェア総合テスト仕様書・報告書 (SWP6)
本ドキュメントは「テスト仕様書」と「テスト報告書」を兼ねる。**まずテスト仕様書としてテスト項目を設計・記入**し、テスト実行後に**結果をチェックしてテスト報告書として完成**させる。
* **1. 導入:** 目的、対象プログラム、テスト環境、参照ドキュメント（SW105要求仕様書、SW205アーキテクチャ設計書）、用語定義
* **2. テスト方針:** テスト戦略（適格性確認テストの範囲と方法）、テスト区分（機能テスト／非機能テスト）、データ準備方針
* **3. 要件対応マトリクス（トレーサビリティ）:** 要求仕様書（SW105等）の**全要件ID**と、対応するテストケースIDの対応表（Markdown表）。全要件が漏れなくカバーされていることを示す。
* **4. テストケース一覧（テスト仕様）:** 各テストケースの詳細をMarkdown表で記述。列構成: テストID / 対応要件ID / テスト区分 / テスト目的 / 前提条件 / 手順 / 期待結果 / **結果** / 備考。**※作成時点（テスト仕様書の段階）では「結果」列は空欄。テスト実行後にチェック印（✅❌等）を記入する。**
* **5. テスト結果総括（テスト報告）:** 合格率サマリー（合格数/全数）、残課題一覧、推奨事項。**※テスト実行後に記入する。**
* **6. 合否判定基準:** 完了定義（例: 全機能要件テスト合格、重大な不合格項目なし等）

## 4. 自己校正のレビュー基準 (Self-Correction Criteria)
エージェントはレビュアーペルソナとして自己精査（Self-Correction）する際、以下の観点でドキュメントを評価する。
1. **完全性:** 各ドキュメントの標準目次構造が網羅されているか。
2. **一貫性:** 要求(SRS) → 設計(Arch) → テスト(Test) の間で内容に矛盾や飛躍がないか。
3. **検証可能性:** 特に要求仕様と設計において、テストケースに落とし込める明確な条件が定義されているか。
