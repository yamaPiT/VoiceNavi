# SWP: ソフトウェア・エンジニアリング・プロセス

[cite_start]ソフトウェア・エンジニアリング・プロセスは、組込みソフトウェア開発に関する作業の中でも、直接ものづくりに関係するソフトウェアとしての要求定義からソフトウェア総合テストまでの作業を定義しています。[cite: 967]

## SWP1 ソフトウェア要求定義

[cite_start]当該製品を実現するためにソフトウェアとして実現が必要となる要求を明確にする。[cite: 974]

### タスク構成と入出力

**【入力】**
* [cite_start]製品企画書[cite: 978]
* (SY106) [cite_start]システム要求仕様書[cite: 979]
* (SY205) [cite_start]システム・アーキテクチャ設計書[cite: 980]
* (SA104) [cite_start]安全要求仕様書[cite: 981]
* [cite_start]ハードウェア仕様書[cite: 982]

**【タスク構成】**
* [cite_start]**SWP1.1 ソフトウェア要求仕様書の作成**[cite: 984]
  * [cite_start]1.1.1 制約条件の確認[cite: 984]
  * [cite_start]1.1.2 ソフトウェア機能要求事項の明確化[cite: 985]
  * [cite_start]1.1.3 ソフトウェア非機能要求事項の明確化[cite: 985]
  * [cite_start]1.1.4 要求の優先順位付け[cite: 985]
  * [cite_start]1.1.5 ソフトウェア要求仕様書の作成[cite: 986]
* [cite_start]**SWP1.2 ソフトウェア要求仕様の確認**[cite: 987]
  * [cite_start]1.2.1 ソフトウェア要求仕様書の内部確認[cite: 988]

**【出力】**
* (SW105) [cite_start]ソフトウェア要求仕様書[cite: 989]
* (SW106) [cite_start]内部確認レポート(ソフトウェア要求定義)[cite: 990, 991]

### 解説
このアクティビティは
* (1.1.1) [cite_start]システム要求仕様とハードウェア仕様をもとに、ソフトウェア要求仕様を検討する際の制約条件を確認した上で、[cite: 996, 997]
* (1.1.2) [cite_start]ソフトウェアに必要となる機能面の要求を明確にし、[cite: 999]
* (1.1.3) [cite_start]また、ソフトウェアに必要な非機能面の要求も合わせて明確にする。[cite: 1001]
* (1.1.4) [cite_start]実際のソフトウェア開発では開発期間やリソース面の制約、あるいは、(1.1.1) で整理したシステムとしての制約などがあるため、これらを考慮したうえで、個々の機能、非機能要求に優先順位をつけていく。[cite: 1003, 1004]
* (1.1.5) [cite_start]これらの検討結果を整理し、ソフトウェア要求仕様書を作成する。[cite: 1006]
* (1.2.1) [cite_start]作成したソフトウェア要求仕様書はあらかじめ確認ポイントを決めて確認を行い、確認した内容を整理して内部確認レポートを作成する。[cite: 1008]

### 留意事項
[cite_start]◆ソフトウェア要求定義の開始条件として以下の事項に留意する。[cite: 1010]
* [cite_start]**製品企画:** 製品戦略(エンドユーザニーズなど)が明確になっている[cite: 1011]
* [cite_start]**製品仕様:** 取扱説明書レベルの内容は決まっている[cite: 1012]
* [cite_start]**スケジュール:** 全体スケジュールは決まっている(発売日、対外的なマイルストーンなど)[cite: 1013, 1014]
* [cite_start]**システム・アーキテクチャ:** 以下の事項が明確になっている[cite: 1015]
  * [cite_start]ハードとソフトの機能分担(ソフトウェア要求は明確化されている)、ハードウェア構成、外部インタフェース、要求性能の実現方法、保守機能、セキュリティ[cite: 1016, 1017, 1018]
* [cite_start]**前提条件:** 使用するソフトウェア (OS、ライブラリなど)、既存製品の利用[cite: 1021]

[cite_start]◆ソフトウェア要求定義にあたっては、システム要求定義で検討した以下の事項に留意する。[cite: 1022]
* [cite_start]組込みシステムの場合、システムが動作する環境など外部環境の分析とそれに伴う異常処理対応の機能分析を考慮する[cite: 1022]
* [cite_start]機能的な側面のみでなく性能や保守性など非機能的な側面を考慮する[cite: 1028]
* [cite_start]対象システムの外部で連携動作するシステムにも留意する[cite: 1029]
* [cite_start]製品として見た場合のロングレンジの製品機能のあり方も考慮する[cite: 1030]

# SWP1 ソフトウェア要求定義（タスク詳細）

## 1.1.1 制約条件の確認

ソフトウェアに関する要求事項を検討するにあたっての考慮すべき制約条件を明確にし、制約条件リストとしてまとめる。

**【入出力と参照情報】**
* **入力:** 製品企画書、(SY106) システム要求仕様書、(SY205) システム・アーキテクチャ設計書、(SA104) 安全要求仕様書、各ハードウェア仕様書
* **出力:** (SW101) 制約条件リスト
* **参照情報:** ISO/IEC 9126-1 (ソフトウェア品質モデル)、ISO 9241 (ユーザビリティ)、ISO 13407 (人間中心設計プロセス)、IEC 61508 (安全性)

**【実施内容 (エージェントの検証項目)】**
以下の事項を確認し、制約条件リストとして整理すること。
1. **製品企画、製品開発戦略:** 特徴的な新機能の有無、プロダクトライン開発の適用など。
2. **製品特性:** 信頼性・安全性要求、耐用年数、想定される利用環境、準拠すべき規格・規約。
3. **ステークホルダの確認:** サービス部門、営業、ハード開発部門などの関係者、およびエンドユーザの特徴。
4. **製品構成の確認:** ハードウェア構成と制約、利用するOS・ミドルウェアの制約、連携システムとのインタフェース。
5. **再利用ソフトウェア:** 既存ソフトウェアの利用有無と、その仕様・制約。
6. **開発・テスト・導入環境:** 開発ツール、テスト環境・ツール、インストール時の制約。

---

## 1.1.2 ソフトウェア機能要求事項の明確化

システムとして実現・提供する機能のうち、ソフトウェアで実現する機能を明確にし、ソフトウェア機能要求リストを作成する。

**【入出力】**
* **入力:** 製品企画書、システム要求仕様書、システム・アーキテクチャ設計書、安全要求仕様書、各ハードウェア仕様書
* **出力:** (SW102) ソフトウェア機能要求リスト

**【実施内容と注意点 (エージェントの検証項目)】**
* ソフトウェアとして実現する機能を明確に切り出すこと。関連するハードウェア機能も明確にすること。
* 競合優位性を確保するための機能を検討すること。
* ユースケース分析（シナリオ、アクティビティ図等）を考慮し、要求漏れがないことを確認すること。
* **異常時**を十分考慮していることを確認すること（フェールセーフ機構、故障時のデータ保護など）。

---

## 1.1.3 ソフトウェア非機能要求事項の明確化

システムとしての機能実現に関係する、ソフトウェアの非機能的な側面の要求を明確にし、リストにまとめる。

**【入出力】**
* **入力:** 製品企画書、システム要求仕様書、システム・アーキテクチャ設計書、安全要求仕様書、各ハードウェア仕様書
* **出力:** (SW103) ソフトウェア非機能要求リスト

**【実施内容 (エージェントの検証項目: 品質モデルに基づく分類)】**
* **信頼性要求:** 想定外の動作への対応、異常処理方式、最低
