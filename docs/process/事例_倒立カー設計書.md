# 倒立カープログラムアーキテクチャ設計書

## 1. はじめに

### 1.1 本書の目的
本書は、倒立カープログラム要件定義書で定められた開発要件をプログラムとして実現するソフトウェアアーキテクチャを定める。具体的には、以下の基本的な設計を行う。
* 主要データ
* タスク
* ハンドラ

### 1.2 使用概念と用語の定義
* **フィードバック制御:** 制御対象の制御量を、目標値に近づけるための帰還ループを持った制御方式。
* **PWM:** パルス幅を変えることによるモータ制御の方式。
* **PI制御:** 差分と積分値によるフィードバック制御の方式。

## 2. データフロー設計

## 3. データのデッドライン
制御アルゴリズムやセンサの計測、ユーザインタフェースの制約から定まるデッドライン（周期）を定義する。

* **3.1 制御制約:** モータ駆動用データは `200µsec` 周期、モータの目標電流計算用データは `10msec` 周期で処理しなければならない。
* **3.2 計測制約:** ロータリーエンコーダ積算値はロータリーエンコーダ割り込みの都度（非同期）に求める。ディレクションサブシステムからの走行コマンドは `10msec` 周期で受信する。
* **3.3 ユーザインタフェース制約:** ユーザ操作に対する処理（回転角速度目標値の設定など）のデッドライン周期は `200msec` 程度とする。
* **3.4 初期化制約:** ジャイロセンサ初期値や初期車速値は、起動時（初期化時）に設定する。

## 4. モジュール設計
データ設定のデッドラインを参考とし、処理をまとめ、3種類のタスクと3種類のハンドラに分割する。

### 4.1 タスク設計
タスクの優先度は、高い順に以下の通りとする。
1. **目標電流算出タスク (優先度: 高):** 10msec周期。モータの目標電流値を求める最重要処理（倒立制御）であり、必ず10msec以内に終える必要があるため最優先とする。
2. **操作指示解釈タスク (優先度: 中):** 200msec周期。ユーザからの操作指示（走行コマンド）を取り扱う処理。
3. **操作指示受信タスク (優先度: 低):** 常時実行。ディレクションサブシステムからの操作指示を読み取る処理。処理時間が極めて短いため、空き時間に実行する。

### 4.2 ハンドラ設計
ハンドラの割り込み優先度は、高い順に以下の通りとする。
1. **目標電流算出タスク起動用周期ハンドラ (優先度: 高):** 10msec周期に目標電流算出タスクを起動する。正確に周期を刻むため最高優先度とする。
2. **ロータリーエンコーダハンドラ (優先度: 中):** 割り込みの都度非同期に起動。正確に回転数を計数するために高い優先度を与える。
3. **モータフィードバック制御ハンドラ (優先度: 低):** 200µsec周期。フィードバック制御処理には比較的長い時間がかかるため、優先度を下げて他の多重割り込みを許可する。

## 5. 非機能要件への対応

| ID | 非機能要件名 | 品質特性 | 設計仕様への対応 |
| :--- | :--- | :--- | :--- |
| NF001 | 納入時にバグがないこと | 信頼性 | 総合テスト仕様書に記載する全テスト項目の合格を確認した後に納入する。 |
| NF002 | パラメータの容易な調整 | 使用性 | パラメータの位置を指示するドキュメントを用意する。 |
| NF003 | ROM/RAM容量 | 効率性 | ROMは126KB、RAMは11.8KB内に収まること。 |
| NF005 | 制御アルゴリズムの変更対応 | 保守性 | 倒立制御処理部分の関数仕様書を作成する。 |
| NF006 | CPUボードの変更 | 移植性 | μITRON上のアプリとして開発。ハードウェア依存データ（ポート番号等）はマクロ定義を行う。 |
